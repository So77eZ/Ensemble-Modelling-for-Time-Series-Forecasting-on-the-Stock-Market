# Анализ проекта: плюсы, минусы, рекомендации

## Плюсы текущего решения

### Архитектурные
- **Стэкинг-ансамбль LSTM + XGBoost** — LSTM улавливает временные зависимости и нелинейные паттерны в последовательностях; XGBoost хорошо работает с плоскими признаками и устойчив к выбросам. Meta-Learner объединяет их оптимально.
- **Квантильная регрессия для CI** — три квантильные модели (α=0.05, 0.50, 0.95) дают статистически обоснованные доверительные интервалы, а не просто ±N%.
- **Walk-forward валидация** — правильный способ кросс-валидации для временных рядов. Предотвращает утечку будущих данных (data leakage), в отличие от случайного разбиения.
- **Bayesian hyperparameter search (Optuna)** — эффективнее grid/random search. Персистентность результатов в JSON позволяет не перезапускать оптимизацию при повторных прогонах.
- **Режим бэктеста** — возможность честно проверить качество модели на прошлых данных, сравнивая прогноз с известным результатом.
- **Два источника данных** — `moexalgo` как основной, прямой MOEX ISS API как fallback. Повышает отказоустойчивость загрузки данных.
- **Обогащение фундаментальными данными** — интеграция с Tinkoff Invest API добавляет P/E, P/B, ROE и другие коэффициенты.
- **Конфигурация через .env** — гиперпараметры вынесены из кода, легко менять без правки скрипта.
- **Два режима запуска** — интерактивный (с подсказками) и CLI (для автоматизации и скриптов).
- **Сохранение всех артефактов** — логи, графики, гиперпараметры, результаты бэктестов структурированы по директориям.

## Минусы и проблемы

### Критические

**1. Ошибка отступа в точке входа (строка 1188)**
```python
# Этот блок находится НЕ внутри if __name__ == '__main__':
if backtest_mode:
    ...
else:
    ...
```
`if backtest_mode:` / `else:` (основной запуск модели) расположены на уровне модуля, вне `if __name__ == '__main__':`. При импорте скрипта этот код выполнится, что вызовет `NameError`. Это структурная ошибка, которая не проявляется только потому, что скрипт запускается напрямую и переменные уже определены выше.

**2. SSL-верификация отключена (`verify=False`)**
```python
resp = requests.post(url, ..., verify=False)
```
Делает все HTTPS-соединения с Tinkoff API уязвимыми к MITM-атакам. Также генерирует `InsecureRequestWarning` в логах. Нужно либо добавить сертификат, либо использовать `certifi`.

**3. XGBoost Optuna оптимизирует на train-данных**
```python
model.fit(X_train, y_train)
preds = model.predict(X_train)  # in-sample!
return mean_squared_error(y_train, preds)
```
Optuna оценивает качество XGBoost на тех же данных, на которых обучает. Это приведёт к выбору параметров с максимальным переобучением (деревья запомнят обучающую выборку). Нужна хотя бы валидационная выборка.

### Качество модели

**4. Большинство фундаментальных признаков = 0**

В `_map_to_model_features` из 17 фундаментальных колонок заполняются только 6 (из Tinkoff API), остальные 11 захардкожены нулём:
```python
'roa': 0.0, 'debt_equity': 0.0, 'current_ratio': 0.0,
'gross_profit_margin': 0.0, 'eps_growth': 0.0, 'sales_growth': 0.0,
'operating_margin': 0.0, 'net_profit_margin': 0.0,
'ps_ratio': 0.0, 'price_cash_flow': 0.0
```
Эти колонки не несут сигнала и создают шум, затрудняя обучение. Константный признак (всегда 0) не даёт модели никакой информации.

**5. Курс USD/RUB захардкожен**
```python
'value_usd': 82.91
```
Жёстко заданный курс превратится в некорректный признак при изменении курса. Курс нужно загружать динамически (ЦБ РФ предоставляет XML API, и `xml.etree.ElementTree` уже импортирован в проекте, но не используется).

**6. Наивная авторегрессия при прогнозе на 2–3 дня**

При прогнозе на день 2 новая строка создаётся как:
```python
'Open': prev_close,
'High': max(prev_close, pred_close) + volatility * 0.3,
'Low':  min(prev_close, pred_close) - volatility * 0.3,
'Volume': temp_df['Volume'].iloc[-1]  # просто копируется
```
Open/High/Low синтетически генерируются по простой формуле, Volume копируется без изменений — это нереалистично и снижает качество 2-х и 3-х дневных прогнозов.

**7. Дубликат логики в `update_technical_indicators_single_row`**

Функция для обновления индикаторов одной строки полностью дублирует логику батчевого `update_technical_indicators`. При изменении любого индикатора нужно менять оба места — источник ошибок.

**8. Прогнозные даты не учитывают торговый календарь**
```python
forecast_dates = [base_date + timedelta(days=i + 1) for i in range(3)]
```
`timedelta(days=1)` может попасть на выходной или праздник. В режиме бэктеста есть обходное решение (поиск ближайшего торгового дня), но в обычном режиме прогноза дата указывается некорректно.

### Производительность и ресурсы

**9. Нет управления памятью GPU**

TensorFlow по умолчанию занимает всю видеопамять. Без `tf.config.experimental.set_memory_growth(gpu, True)` при наличии GPU возможны OOM-ошибки при параллельном использовании ресурсов.

**10. Медленная валидация: 3 независимых обучения**

При walk-forward обучаются 3 полных ансамбля (LSTM + 4 XGBoost + meta-learner = 6 моделей × 3 сплита = 18 моделей). При использовании Optuna сверху ещё N_TRIALS обучений. На CPU с LSTM это очень медленно.

### Код и качество

**11. Мёртвый код и неиспользуемые импорты**
- `TensorBoard` импортируется (`from tensorflow.keras.callbacks import TensorBoard`), но не используется нигде в коде.
- `scipy.interpolate.interp1d` импортируется, но не используется.
- `TINVEST_USE_GRPC` в `.env` — не используется в v13.
- `FINANCEMARKER_TOKEN` в `.env` — не используется.

**12. Мета-learner не оптимизируется через Optuna**

Гиперпараметры мета-леарнера захардкожены:
```python
meta_params = {'n_estimators': 100, 'max_depth': 6, 'learning_rate': 0.03, ...}
```
При этом базовые модели оптимизируются Optuna. Непоследовательно.

**13. Дата начала данных жёстко задана**
```python
start_date = '2014-01-01'
```
Не настраивается через `.env` или CLI. Для тикеров с короткой историей (IPO после 2014) загружается пустой период в начале; для коротких экспериментов всегда грузятся 10+ лет данных.

**14. Логирование raw API response**
```python
logger.info(f"RAW API RESPONSE: {resp.text[:500]}...")
```
Первые 500 символов ответа API пишутся в лог-файл. Ответ может содержать токены или чувствительные данные.

## Что можно добавить / улучшить

### Высокий приоритет (влияют на качество)

1. **Правильная оценка XGBoost в Optuna** — использовать `train_test_split` на данных оптимизации (например, последние 10% как holdout), чтобы Optuna не выбирала переобученные параметры.

2. **Заполнить нулевые фундаментальные признаки** — либо найти API-источник (например, Smart-Lab, Финансовый Маркер, уже есть токен), либо удалить пустые колонки из вектора признаков.

3. **Динамический курс USD/RUB** — использовать уже импортированный `xml.etree.ElementTree` с API ЦБ РФ (`http://www.cbr.ru/scripts/XML_daily.asp`).

4. **Торговый календарь** — фильтровать прогнозные даты, исключая выходные и праздники MOEX (библиотека `exchange_calendars` или `pandas_market_calendars`).

### Средний приоритет (архитектура)

5. **Прямое многошаговое прогнозирование** — вместо авторегрессии обучить отдельные модели для каждого горизонта (1d, 2d, 3d). Это избегает накопления ошибки и устраняет необходимость в синтетическом Open/High/Low/Volume.

6. **Рефакторинг дублированной логики индикаторов** — объединить `update_technical_indicators` и `update_technical_indicators_single_row` в одну функцию, принимающую параметр `mode`.

7. **Исправить отступ `if backtest_mode:`** — перенести в тело `if __name__ == '__main__':`.

8. **Фиксация SSL** — убрать `verify=False`, добавить `certifi` или корректный пропуск с предупреждением только при необходимости.

### Низкий приоритет (удобство)

9. **`requirements.txt` или `pyproject.toml`** — воспроизводимость окружения (сейчас зависимости нигде не зафиксированы).

10. **GPU memory growth** — добавить в инициализацию:
    ```python
    for gpu in tf.config.list_physical_devices('GPU'):
        tf.config.experimental.set_memory_growth(gpu, True)
    ```

11. **Configurable start date** — добавить переменную `DATA_START_DATE` в `.env` и аргумент `--start-date` в CLI.

12. **Оптимизация мета-леарнера** — включить его гиперпараметры в Optuna или хотя бы вынести в константы/env.

13. **Расширение горизонта прогноза** — сделать горизонт настраиваемым через `.env` или CLI (сейчас жёстко задано [1, 2, 3]).

14. **Прогресс-бар для Optuna** — вернуть `show_progress_bar=True` или использовать tqdm, чтобы пользователь видел прогресс долгой оптимизации.
